Sprints & User Stories

Sprint 1 (Auth & Setup)

User Registration (5 pts)

High

As a student, I want to register with my valid college email to securely access the marketplace.

Acceptance Criteria:
  • Validate only specific college email domains.
  • Secure password hashing via Bcrypt.
  • Block duplicate email registrations.

Browse Feed (3 pts)

High

As a user, I want to view all available products clearly on the dashboard.

Acceptance Criteria:
  • Fetch data via GET /api/products.
  • Display responsive grid using CSS Grid.
Sprint 2 (Core Logic)

Demand Queue (13 pts)

Critical

As a buyer, I want to express interest in an item and be placed in a fair queue.

Acceptance Criteria:
  • Implement strict First-In-First-Out logic.
  • Prevent users from clicking "Interest" multiple times.
  • Process clicks within 200ms to avoid concurrency conflicts.

Product Posting (8 pts)

High

As a seller, I want to post my academic items with a price and description.

Acceptance Criteria:
  • Form validation to reject negative prices.
  • Secure upload pipeline for product imagery.

Detailed Story Analysis

User Authentication Analysis

Sprint: 1 | Points: 5 | Dev Team
Task Breakdown
  • Backend: API /auth/register, input validation, JWT issuance.
  • Frontend: Responsive login forms, React state management for tokens.
Testing Highlights
  • Negative: Reject invalid email domains and wrong passwords.
  • Performance: Handle simultaneous login requests efficiently.

Demand Queue Implementation

Sprint: 2 | Points: 13 | Lead / DB
Task Breakdown
  • Backend: Build queue endpoints, implement transactional locks to prevent race conditions.
  • DB: Add index on timestamp for rapid FIFO sorting.
Testing Highlights
  • Positive: Queue order strictly maintained based on exact millisecond of click.
  • Edge Case: Handled scenario where two buyers click exactly simultaneously.

Non-Functional Requirements

Performance

System must process "Interested" clicks within 200ms to guarantee queue integrity.

Security

Prevent unauthorized access and protect user data via Bcrypt encryption and JWT.

Scalability

Designed to handle thousands of concurrent student users across the campus.

Reliability & Fairness

Maintain 99% uptime. Allocate items strictly based on FIFO rules without manipulation.

Functional Requirements Summary

  • Register and login via secure interface using valid credentials.
  • Provide form interface to add, edit, and delete product listings.
  • Display dynamic grid of available products on the home page.
  • Execute searches to find products based on specific keywords.
  • Filter and sort products based on category and price brackets.
  • Allow users to create, view, edit, and delete personal profiles.
